<architectural_notes>
## Single ARQ worker rationale (CONTEXT 已解决的开放问题 O-5, 2026-05-19)

Health check cron runs on the **existing Phase 4 ARQ worker** (NOT a separate worker process). Justification:

- **Async + semaphore-limited:** `HealthCheckService.run_health_checks` uses `asyncio.Semaphore(HEALTH_CHECK_CONCURRENCY=5)` so concurrent upstream probes never exceed 5 in-flight HTTP requests.
- **N-hourly cadence:** the source cron is `cron(minute={0, 10, 20, 30, 40, 50})` — six runs per hour, each completing in seconds, so the cumulative duty cycle on the worker is well under 1%.
- **Zero hot-path impact:** ARQ workers are separate processes from the uvicorn web workers, so even if a health check runs long, the relay path is unaffected.
- **2h4g deployment constraint:** spinning up a second ARQ worker would cost ~350MB resident memory (per CLAUDE.md `Constraints`) — unnecessary at this workload.

This is the architectural justification for Task 3 wiring `run_health_checks` into the **existing** `WorkerSettings.functions` + `cron_jobs` rather than introducing a dedicated worker process.
</architectural_notes>
