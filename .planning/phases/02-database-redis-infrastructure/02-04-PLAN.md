# Plan 02-04: Alembic init and baseline migration

<objective>
为 api-service 初始化 Alembic 迁移基础设施，创建包含所有 22 张表最终 DDL 的 baseline 迁移文件。该迁移合并 user-service（9 表）和 admin-service（13 表）的最终表结构，作为 api-service 迁移链的起点（revision 1）。同时创建 `api_service/models/__init__.py` 占位文件供 `_env_shared.py` 的 `_load_metadata` 导入。
</objective>

<must_haves>
## Goal-Backward Verification

**truths** (must be true after execution):
- `services/api-service/migrations/alembic.ini` 存在，`service_name = api-service`，`service_package = api_service`，`database_env = DATABASE_URL`
- `services/api-service/migrations/env.py` 调用 `api_service.common.infra.db._env_shared.run_env()`
- baseline 迁移 revision 为 `"20260519_baseline"`，`down_revision = None`
- baseline 包含所有 22 张表的 `CREATE TABLE IF NOT EXISTS` DDL
- 所有金额字段使用 BIGINT（微元精度）
- 包含 seed data（model_vendors, model_categories, audit_action_definitions）
- `api_service/models/__init__.py` 存在（可为空或仅含 docstring）
- `alembic -c migrations/alembic.ini history` 显示一个 revision

**anti-goals** (must NOT happen):
- 不得遗漏任何表（必须覆盖两库合并后的全部表）
- 不得使用旧的列名（必须反映所有重命名迁移的最终状态）
- 不得遗漏 CHECK 约束、FK 约束、唯一索引
- 不得包含 `alembic_version` 表的 DDL（Alembic 自动管理）
- downgrade 不得遗漏任何表的 DROP（按 FK 依赖逆序）
</must_haves>

创建 `services/api-service/migrations/versions/20260519_baseline.py`：

文件结构：
- `revision = "20260519_baseline"`
- `down_revision = None`
- `branch_labels = None`
- `depends_on = None`

`upgrade()` 函数包含所有 22 张表的 `CREATE TABLE IF NOT EXISTS` DDL，使用 `op.execute("""...""")` 执行原始 SQL。

表必须按 FK 依赖顺序创建（被依赖的表先创建）：

**User-Service 表（9 张，最终状态）：**
1. `users` — 含 balance/frozen_amount/used_amount BIGINT, rpm_limit INT DEFAULT 20, record_ip_log TINYINT(1) DEFAULT 0
2. `email_verification_codes` — FK → users(id)
3. `user_sessions` — FK → users(id), 含 expires_at 索引
4. `user_api_keys` — FK → users(id), 含 quota_limit/quota_used BIGINT
5. `balance_transactions` — FK → users(id), 含 amount/balance_before/balance_after BIGINT
6. `topup_orders` — FK → users(id), 含 amount BIGINT
7. `api_call_logs` — FK → users(id) + user_api_keys(id), 重构后 14 核心列 + log_type + other JSON, quota BIGINT
8. `usage_stats` — FK → users(id)
9. `voucher_redemption_codes` — 含 amount BIGINT, redeemed_by FK → users(id) nullable

**Admin-Service 表（13 张，最终状态）：**
10. `admin_users` — role SMALLINT, status SMALLINT, created_by FK → self nullable
11. `audit_action_definitions` — 含 updated_at, updated_by
12. `admin_audit_logs` — FK → admin_users(id) nullable, FK → audit_action_definitions(action_key)
13. `model_vendors` — 独立表
14. `model_categories` — 独立表
15. `model_catalog` — FK → model_vendors(id), 含 sale_input_price/sale_output_price
16. `model_catalog_category_map` — FK → model_catalog(id) + model_categories(id)
17. `routing_configs` — FK → admin_users(id) nullable
18. `provider_credentials` — 独立表
19. `routing_settings` — FK → admin_users(id) nullable (updated_by)
20. `pools` — FK → admin_users(id) nullable (created_by SET NULL)
21. `pool_model_configs` — FK → pools(id) + model_catalog(id), 含 cost_input_price/cost_output_price
22. `pool_accounts` — FK → pools(id) + provider_credentials(id), status SMALLINT 0-3, balance CHECK >= 0, created_by FK → admin_users(id) nullable

每张表必须包含：
- 完整的列定义（类型、DEFAULT、COMMENT、NOT NULL）
- PRIMARY KEY
- UNIQUE KEY（如有）
- INDEX（如有）
- FOREIGN KEY 约束（如有）
- CHECK 约束（如有）
- ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci

`upgrade()` 末尾包含 seed data INSERT：
- `model_vendors` 初始数据
- `model_categories` 初始数据
- `audit_action_definitions` 初始数据

`downgrade()` 函数按 FK 依赖逆序 DROP TABLE IF EXISTS 所有 22 张表。

**关键注意事项：**
- 必须读取所有后续迁移文件确认最终列名和类型
- 金额字段全部为 BIGINT（微元），不是 DECIMAL
- `model_catalog` 不是 `supported_models`（已重命名）
- `pool_model_configs` 不是 `pool_models`（已重命名）
- `model_catalog_category_map` 不是 `supported_model_category_map`（已重命名）
- 价格列使用 `sale_input_price`/`sale_output_price`/`cost_input_price`/`cost_output_price`（已重命名）
- `api_call_logs` 使用重构后的 14 核心列结构（20260518_01 迁移）
- `users` 表包含 `record_ip_log` 列（20260518_02 迁移）
