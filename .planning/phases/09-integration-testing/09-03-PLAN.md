Part A — Memory test (per D-09, D-10):

`test_four_workers_memory_under_limit`:
- Mark with `@pytest.mark.slow` (per D-10).
- Spawn uvicorn with 4 workers: `subprocess.Popen(["python", "-m", "uvicorn", "api_service.main:app", "--host", "127.0.0.1", "--port", "9999", "--workers", "4"])`. Set env vars: DATABASE_URL, REDIS_URL, CACHE_REDIS_URL, JWT_SECRET_KEY, INTERNAL_SECRET, INFERENCE_SERVICE_URL, BOOTSTRAP_SUPERADMIN_ENABLED=false, BOOTSTRAP_SUPERADMIN_REQUIRE_ON_STARTUP=false. Use cwd=`services/api-service` directory.
- Wait for startup: poll `http://127.0.0.1:9999/health` with httpx every 1s, timeout 30s.
- Send 20 concurrent requests to `/v1/models` (with valid API key header) using asyncio.gather to warm up workers.
- Measure memory: `parent = psutil.Process(proc.pid)`, `children = parent.children(recursive=True)`, `total_rss = sum(c.memory_info().rss for c in children) + parent.memory_info().rss`.
- Assert `total_rss < 1.5 * 1024**3` (1.5GB).
- Log actual RSS for visibility: `print(f"Total RSS: {total_rss / 1024**2:.1f} MB")`.
- Cleanup: `proc.terminate()`, `proc.wait(timeout=10)`.
- Wrap in try/finally to ensure process cleanup.

Part B — Snowflake ID concurrency tests (per D-11, D-12):

`test_snowflake_worker1_no_collision`:
- Configure snowflake with worker_id=1, datacenter_id=1.
- Launch 4 coroutines each generating 10000 IDs via `generate_snowflake_id()`.
- Collect all 40000 IDs into a set.
- Assert `len(id_set) == 40000` (zero collisions).

`test_snowflake_worker2_no_collision`:
- Same as above but with worker_id=2, datacenter_id=1.
- 4 coroutines x 10000 IDs = 40000 unique.

`test_snowflake_cross_worker_no_collision` (per D-12):
- Generate 10000 IDs with worker_id=1, datacenter_id=1.
- Generate 10000 IDs with worker_id=2, datacenter_id=1.
- Combine into one set.
- Assert `len(combined) == 20000` (no cross-worker collision).

For the coroutine-based tests, use `asyncio.gather` with 4 tasks. Each task runs a loop calling `generate_snowflake_id()` synchronously (it's a fast in-memory operation). Use `asyncio.to_thread` or just run in the event loop since SnowflakeGenerator is not truly async — the point is to test concurrent access patterns.

Actually, since `generate_snowflake_id` uses a module-level `@lru_cache` singleton, concurrent coroutines in the same event loop will serialize on the GIL anyway. The real concurrency test is: call it rapidly from multiple coroutines to verify the generator's internal sequence counter handles rapid successive calls without producing duplicates. Use `loop.run_in_executor` with a ThreadPoolExecutor(max_workers=4) to get true thread-level concurrency for a stronger test.

Import: `from api_service.common.utils.snowflake import configure_snowflake, generate_snowflake_id, get_snowflake_generator, SnowflakeIDGenerator`.

Reset the generator cache between tests: call `get_snowflake_generator.cache_clear()` in a fixture or at test start.
