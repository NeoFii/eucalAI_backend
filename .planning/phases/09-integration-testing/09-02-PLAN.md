  <action>
Create cross-domain integration tests per D-13 through D-15. These tests verify the full admin-write -> Redis version INCR -> RoutingConfigCache reload -> relay behavior change pipeline.

Tests use the `db_session`, `cache_redis`, `session_factory`, and `init_relay` fixtures from conftest.py (Plan 09-01). The RoutingConfigCache instance is accessed via `get_routing_config_cache()` from `api_service.relay.dependencies`.

Implementation approach per D-14: After admin writes, call `await config_cache.check_and_reload(session_factory)` directly to force version check without waiting for poll interval. This simulates what happens on the next relay request.

Tests to implement:

1. `test_add_model_route_propagates` (per D-15 scenario 1):
   - Setup: seed_routing_config has "gpt-4o-mini" routed. Verify config_cache.load() contains "gpt-4o-mini" in model_channels.
   - Action: Insert a new ModelCatalog entry "claude-3-haiku" + new PoolModelConfig linking it to existing pool. Call `await get_cache_redis().incr(ROUTING_CONFIG_VERSION_KEY)` (simulating what RoutingSettingService._bump_version does after admin write).
   - Verify: Call `await config_cache.check_and_reload(session_factory)`. Assert config_cache.load() now contains "claude-3-haiku" in model_channels or model_providers.
   - Cleanup: Transaction rollback handles it.

2. `test_disable_channel_propagates` (per D-15 scenario 2):
   - Setup: seed_routing_config has one active Pool (status=1). Verify config_cache.load() has at least one channel.
   - Action: Update Pool.status = 0 (disabled) in db_session. Flush. Bump Redis version key.
   - Verify: Call `await config_cache.check_and_reload(session_factory)`. Assert the disabled pool no longer appears in config_cache.load() model_channels for the affected model.

3. `test_modify_price_propagates` (per D-15 scenario 3):
   - Setup: seed_routing_config has ModelCatalog with input_price=X. Record original price from config_cache.load().
   - Action: Update ModelCatalog.input_price to new value in db_session. Flush. Bump Redis version key.
   - Verify: Call `await config_cache.check_and_reload(session_factory)`. Assert config_cache.load() model_prices for "gpt-4o-mini" reflects the new input_price.

4. `test_version_unchanged_no_reload`:
   - Setup: Record current config_cache._version.
   - Action: Do NOT bump Redis version key. Call `await config_cache.check_and_reload(session_factory)`.
   - Verify: Assert config_cache._version unchanged (no unnecessary DB reload).

5. `test_redis_version_increment_on_admin_write`:
   - Setup: Read current version from Redis: `await cache_redis.get(ROUTING_CONFIG_VERSION_KEY)`.
   - Action: Call `RoutingSettingService._bump_version()`.
   - Verify: Read version again. Assert it incremented by 1.

Import ROUTING_CONFIG_VERSION_KEY from `api_service.relay.config_cache` (or `api_service.services.admin.routing_setting_service`). Access the RoutingConfigCache singleton via `get_routing_config_cache()`.

Note: For tests 1-3, the db_session uses transaction rollback, but `check_and_reload` opens its own session via session_factory. To make the DB changes visible to the reload, these tests need to COMMIT the changes before calling check_and_reload. Use a separate approach: instead of relying on the rollback fixture for data isolation, these tests should insert data, commit, run assertions, then DELETE the inserted rows in a finally block. Alternatively, use `await db_session.commit()` and accept that these specific tests need manual cleanup (the conftest rollback won't help since check_and_reload uses its own session). Document this tradeoff in a comment.

Simpler approach: For tests 1-3, bypass the transaction-rollback db_session. Instead, use session_factory directly to create a committed session, run the test, then clean up. The `cache_redis.flushdb()` in session teardown handles Redis cleanup.
